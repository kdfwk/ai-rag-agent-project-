## 1. 项目概述

### 1.1 这个项目是做什么的？
这是一个**基于 ReAct 模式的扫地机器人智能客服系统**，主要功能：
- 接收用户关于扫地机器人的咨询问题
- 通过 RAG（检索增强生成）从知识库中检索专业资料
- 调用多种工具获取天气、位置、用户信息等上下文
- 生成个性化的用户使用报告
- 支持流式输出（打字机效果）

### 1.2 技术栈
- **Python 3.x** - 编程语言
- **LangChain** - AI 应用开发框架
- **Chroma** - 向量数据库，存储知识库
- **Streamlit** - Web 界面框架
- **Qwen3-Max** - 阿里通义千问大语言模型
- **text-embedding-v4** - 文本嵌入模型
- **PyYAML** - YAML 配置文件解析

### 第五步：文件处理工具（utils/file_handler.py）

**文件位置**：`utils/file_handler.py`

```python
def get_file_md5_hex(filepath: str):
    """获取文件的md5的十六进制字符串"""
    md5_obj = hashlib.md5()
    chunk_size = 4096  # 4KB分片，避免文件过大爆内存
    with open(filepath, "rb") as f:
        while chunk := f.read(chunk_size):
            md5_obj.update(chunk)
    return md5_obj.hexdigest()

def listdir_with_allowed_type(path: str, allowed_types: tuple[str]):
    """返回文件夹内的文件列表（允许的文件后缀）"""
    files = []
    for f in os.listdir(path):
        if f.endswith(allowed_types):
            files.append(os.path.join(path, f))
    return tuple(files)

def pdf_loader(filepath: str) -> list[Document]:
    return PyPDFLoader(filepath).load()

def txt_loader(filepath: str) -> list[Document]:
    return TextLoader(filepath, encoding="utf-8").load()
```

#### 这是在干嘛？
提供文件读取和处理的通用功能。

#### 关键功能解释：

**① MD5 计算**
- **作用**：为每个文件生成唯一标识
- **为什么需要**：避免重复加载相同的文件到向量库
- **分片读取**：大文件不会一次性加载到内存，防止内存溢出

**② 文件类型过滤**
- **作用**：只读取指定类型的文件（txt、pdf）
- **为什么需要**：避免加载无关文件（如图片、视频）

**③ 文档加载器**
- **PyPDFLoader**：解析 PDF 文件，提取文本内容
- **TextLoader**：读取 TXT 文件，保持 UTF-8 编码

```python
class VectorStoreService:
    def __init__(self):
        self.vector_store = Chroma(

        self.spliter = RecursiveCharacterTextSplitter(

            chunk_overlap=chroma_conf["chunk_overlap"],
            separators=chroma_conf["separators"],
            length_function=len,
        )
```

#### 这是在干嘛？
管理向量数据库，负责知识库的构建和检索。

#### 核心组件解释：

**① Chroma 向量库**
```python
self.vector_store = Chroma(...)
```
- **作用**：存储文本向量，支持语义搜索
- **collection_name**：类似数据库的表名
- **persist_directory**：向量数据持久化保存的目录
- **embedding_function**：用于将文本转换为向量的模型

**② 文本分片器**
```python
self.spliter = RecursiveCharacterTextSplitter(...)
```
- **作用**：把长文档切分成小片段
- **chunk_size=200**：每个片段最多 200 个字符
- **chunk_overlap=20**：相邻片段重叠 20 个字符，保持上下文连贯
- **separators**：优先在段落、句子边界处切分

#### 知识库构建流程：

```python
def load_document(self):
    # 1. 扫描数据目录
    allowed_files_path = listdir_with_allowed_type(...)
    
    for path in allowed_files_path:
        # 2. 计算文件 MD5
        md5_hex = get_file_md5_hex(path)
        
        # 3. 检查是否已处理过（去重）
        if check_md5_hex(md5_hex):
            continue
        
        # 4. 读取文件内容
        documents = get_file_documents(path)
        
        # 5. 分片处理
        split_document = self.spliter.split_documents(documents)
        
        # 6. 存入向量库
        self.vector_store.add_documents(split_document)
        
        # 7. 记录 MD5，避免重复处理
        save_md5_hex(md5_hex)
```

#### 为什么要分片？
- **限制长度**：LLM 的上下文窗口有限，不能处理超长文本
- **提高精度**：检索时返回最相关的片段，而不是整个文档
- **节省成本**：只发送必要的信息给模型，减少 token 消耗

### 第八步：RAG 总结服务（rag/rag_service.py）

**文件位置**：`rag/rag_service.py`

```python
class RagSummarizeService(object):
    def __init__(self):
        self.vector_store = VectorStoreService()
        self.retriever = self.vector_store.get_retriever()
        self.prompt_template = PromptTemplate.from_template(self.prompt_text)
        self.model = chat_model
        self.chain = self._init_chain()
    
    def _init_chain(self):
        chain = self.prompt_template | print_prompt | self.model | StrOutputParser()
        return chain
    
    def rag_summarize(self, query: str) -> str:
        # 1. 检索相关文档
        context_docs = self.retriever_docs(query)
        
        # 2. 构建上下文字符串
        context = ""
        for i, doc in enumerate(context_docs):
            context += f"【参考资料{i+1}】: {doc.page_content}\n"
        
        # 3. 调用模型生成回答
        return self.chain.invoke({
            "input": query,
            "context": context,
        })
```

#### 这是在干嘛？
实现 RAG（检索增强生成）的核心逻辑。

#### RAG 工作流程：

**① 检索（Retrieval）**
```python
context_docs = self.retriever_docs(query)
```
- 用户提问："小户型适合哪些扫地机器人"
- 向量库检索：找到 3 个最相关的文档片段
- 原理：计算问题向量与文档向量的相似度

**② 增强（Augmentation）**
```python
context = "【参考资料1】: ...\n【参考资料2】: ...\n"
```
- 把检索到的文档片段组合成上下文
- 格式化为模型能理解的提示词

**③ 生成（Generation）**
```python
self.chain.invoke({"input": query, "context": context})
```
- 将问题和上下文一起发送给 LLM
- LLM 基于参考资料生成准确回答

#### LangChain Chain 解释：
```python
chain = prompt_template | model | StrOutputParser()
```
- **管道式处理**：数据依次经过每个组件
- **prompt_template**：填充提示词模板
- **model**：调用大语言模型
- **StrOutputParser**：解析模型输出为字符串

### 第十步：中间件系统（agent/tools/middleware.py）

```python
@wrap_tool_call
def monitor_tool(request: ToolCallRequest, handler: Callable) -> ToolMessage | Command:
    """工具调用监控"""
    logger.info(f"[tool monitor]执行工具：{request.tool_call['name']}")
    logger.info(f"[tool monitor]传入参数：{request.tool_call['args']}")
    
    try:
        result = handler(request)
        logger.info(f"[tool monitor]工具调用成功")
        
        # 特殊处理：如果是报告上下文注入工具，设置标记
        if request.tool_call['name'] == "fill_context_for_report":
            request.runtime.context["report"] = True
        
        return result
    except Exception as e:
        logger.error(f"工具调用失败：{str(e)}")
        raise e

@before_model
def log_before_model(state: AgentState, runtime: Runtime):
    """模型调用前日志记录"""
    logger.info(f"[log_before_model]即将调用模型，带有{len(state['messages'])}条消息。")
    return None

@dynamic_prompt
def report_prompt_switch(request: ModelRequest):
    """动态切换提示词"""
    is_report = request.runtime.context.get("report", False)
    if is_report:
        return load_report_prompts()  # 报告生成提示词
    return load_system_prompts()      # 普通对话提示词
```

#### 这是在干嘛？
在智能体执行过程中插入自定义逻辑。

**① wrap_tool_call（工具调用包装）**
- **时机**：每次调用工具前后
- **作用**：
  - 记录工具调用日志
  - 捕获异常
  - 设置运行时上下文（如 report 标记）

**② before_model（模型调用前）**
- **时机**：每次调用 LLM 之前
- **作用**：
  - 记录当前对话状态
  - 调试时查看消息历史

**③ dynamic_prompt（动态提示词）**
- **时机**：每次构建提示词之前
- **作用**：
  - 根据上下文切换不同的提示词
  - 实现场景感知的行为变化

#### 上下文传递机制：
```python

# 2. 下次调用 LLM 时，report_prompt_switch 检测到标记
```

### 第十一步：ReAct 智能体（agent/react_agent.py）

**文件位置**：`agent/react_agent.py`

```python
class ReactAgent:
    def __init__(self):
        self.agent = create_agent(
            model=chat_model,
            system_prompt=load_system_prompts(),
            tools=[rag_summarize, get_weather, get_user_location, get_user_id,
                   get_current_month, fetch_external_data, fill_context_for_report],
            middleware=[monitor_tool, log_before_model, report_prompt_switch],
        )
    
    def execute_stream(self, query: str):
        input_dict = {
            "messages": [
                {"role": "user", "content": query},
            ]
        }
        
        # 流式输出，context 用于传递报告标记
        for chunk in self.agent.stream(input_dict, stream_mode="values", 
                                       context={"report": False}):
            latest_message = chunk["messages"][-1]
            if latest_message.content:
                yield latest_message.content.strip() + "\n"
```

#### 这是在干嘛？
创建和运行 ReAct 智能体。

#### ReAct 模式解释：

**ReAct = Reasoning（推理） + Acting（行动）**

智能体的思考流程：
```
用户：我家有宠物，适合用什么扫地机器人？

思考1：用户需要了解宠物家庭的扫地机器人选购建议，我需要查询相关知识
行动1：调用 rag_summarize(query="宠物家庭 扫地机器人 选购")

观察1：返回资料提到需要强吸力、防缠绕刷头...

思考2：信息已经足够，可以生成回答
行动2：生成最终回复
```

#### create_agent 参数说明：

**① model**
- 使用的 LLM（Qwen3-Max）
- 负责推理和生成

**② system_prompt**
- 智能体的"人格设定"
- 定义行为准则和工具使用规则
- 详见 `prompts/main_prompt.txt`

**③ tools**
- 智能体可以调用的工具列表
- LLM 会根据需要自主决定调用哪个工具

**④ middleware**
- 执行过程中的拦截器
- 实现监控、日志、动态行为等功能

#### 流式输出解释：
```python
for chunk in self.agent.stream(...):
    yield latest_message.content.strip() + "\n"
```
- **stream_mode="values"**：返回每次迭代的状态值
- **yield**：逐个返回生成的内容
- **效果**：前端可以实时显示正在生成的文字（打字机效果）

## 5. 关键技术点

### 5.1 ReAct 模式详解

#### 什么是 ReAct？
ReAct 是一种让 LLM 具备推理和行动能力的框架：
- **Reasoning（推理）**：分析问题，决定下一步行动
- **Acting（行动）**：调用工具获取信息
- **Observation（观察）**：查看工具返回结果
- **循环**：重复上述过程直到问题解决

#### 传统 LLM vs ReAct Agent

**传统 LLM**：
```
用户：今天北京天气怎么样？
LLM：（基于训练数据回答，可能是过时的信息）
```

**ReAct Agent**：
```
用户：今天北京天气怎么样？

思考：我需要获取北京的实时天气信息
行动：调用 get_weather(city="北京")
观察：返回"北京天气为晴天，气温26度..."
思考：信息已足够
行动：生成最终回答"今天北京天气晴朗，气温26度..."
```

#### 本项目中的 ReAct 实现
```python
# LangChain 的 create_agent 自动实现了 ReAct 循环
self.agent = create_agent(
    model=chat_model,
    system_prompt=load_system_prompts(),  # 定义思考规则
    tools=[...],                           # 可执行的动作
    middleware=[...]                       # 过程监控
)
```

### 5.2 RAG（检索增强生成）原理

#### 为什么需要 RAG？
- **LLM 的局限性**：
  - 训练数据有时效性（可能不知道最新产品）
  - 容易产生"幻觉"（编造事实）
  - 无法访问私有数据（如用户手册）

- **RAG 的优势**：
  - 基于真实资料回答
  - 可以更新知识库而不重新训练模型
  - 提供引用来源，增加可信度

#### RAG 工作流程
```
1. 用户提问："小户型适合哪些扫地机器人？"
         ↓
2. 使用 embedding 模型将问题转为向量
   [0.12, -0.45, 0.78, ...]
         ↓
3. 在向量库中搜索最相似的文档向量
   - 文档1向量：[0.11, -0.43, 0.76, ...]  相似度 0.95
   - 文档2向量：[0.13, -0.46, 0.79, ...]  相似度 0.92
   - 文档3向量：[0.10, -0.44, 0.77, ...]  相似度 0.90
         ↓
4. 返回对应的文本内容
   - "小户型建议选择宽度<35cm的机型..."
   - "推荐型号：石头T8、科沃斯N9+"
   - "注意避开高度>10cm的门槛..."
         ↓
5. 将问题 + 参考资料发送给 LLM
   Prompt: """
   用户问题：小户型适合哪些扫地机器人？
   
   参考资料：
   【参考资料1】: 小户型建议选择宽度<35cm的机型...
   【参考资料2】: 推荐型号：石头T8、科沃斯N9+...
   【参考资料3】: 注意避开高度>10cm的门槛...
   
   请基于以上资料回答问题。
   """
         ↓
6. LLM 生成准确回答
   "对于小户型，建议选择宽度小于35cm的紧凑型扫地机器人，
   如石头T8和科沃斯N9+。同时要注意家中门槛高度..."
```

#### 向量相似度计算
- **余弦相似度**：计算两个向量夹角的余弦值
- **范围**：-1（完全相反）到 1（完全相同）
- **阈值**：通常 > 0.8 认为高度相似

### 5.3 动态提示词切换

#### 为什么要动态切换？
不同场景需要不同的提示词：
- **普通问答**：强调准确性和专业性
- **报告生成**：强调格式规范和数据分析

#### 实现机制
```python
# 1. 运行时上下文（Runtime Context）
#    类似一个全局字典，在请求生命周期内共享
runtime.context = {"report": False}

# 2. 工具调用时设置标记
@wrap_tool_call
def monitor_tool(request, handler):
    if request.tool_call['name'] == "fill_context_for_report":
        request.runtime.context["report"] = True  # 设置为报告模式
    return handler(request)

# 3. 每次调用 LLM 前检查标记
@dynamic_prompt
def report_prompt_switch(request):
    is_report = request.runtime.context.get("report", False)
    if is_report:
        return load_report_prompts()  # 返回报告提示词
    return load_system_prompts()       # 返回普通提示词
```

#### 时序图
```
用户请求
   ↓
Agent 开始执行 (context={"report": False})
   ↓
LLM 决定调用 fill_context_for_report
   ↓
monitor_tool 拦截 → 设置 context["report"]=True
   ↓
LLM 继续执行，准备下一次调用
   ↓
report_prompt_switch 检测到 report=True
   ↓
加载 report_prompt.txt
   ↓
LLM 基于报告提示词生成内容
```

### 5.4 流式输出原理

#### 什么是流式输出？
- **非流式**：等待 LLM 生成完所有内容，一次性返回
- **流式**：LLM 每生成一个 token（词元）就立即返回

#### 优势对比
| 特性 | 非流式 | 流式 |
|------|--------|------|
| 首字延迟 | 高（需等待全部生成） | 低（立即看到第一个字） |
| 用户体验 | 长时间空白后突然显示 | 打字机效果，更自然 |
| 资源占用 | 需要缓存完整响应 | 边生成边传输 |

#### 实现细节
```python
# 1. Agent 层：启用流式模式
for chunk in self.agent.stream(input_dict, stream_mode="values"):
    latest_message = chunk["messages"][-1]
    if latest_message.content:
        yield latest_message.content.strip() + "\n"

# 2. Streamlit 层：逐字符显示
def capture(generator, cache_list):
    for chunk in generator:
        cache_list.append(chunk)  # 保存完整块
        for char in chunk:        # 逐字符遍历
            time.sleep(0.01)      # 控制速度
            yield char            # 逐个返回字符

st.chat_message("assistant").write_stream(capture(res_stream, response_messages))
```

#### 数据流动
```
LLM 生成: "你" → "好" → "，" → "我" → "是" → "智" → "能" → "客" → "服"
            ↓       ↓       ↓       ↓       ↓       ↓       ↓       ↓       ↓
Agent:    yield "你"  yield "好"  yield "，"  ...
            ↓       ↓       ↓
Streamlit: 显示"你" 显示"你好" 显示"你好，" ...
```

### 5.5 向量数据库去重机制

#### 为什么需要去重？
- 避免重复加载相同文件，浪费时间和存储空间
- 保证知识库的一致性

#### MD5 去重流程
```python
# 1. 计算文件 MD5
md5_hex = get_file_md5_hex(path)
# 例如：d41d8cd98f00b204e9800998ecf8427e

# 2. 检查是否已处理
def check_md5_hex(md5_for_check: str):
    with open("md5.text", "r") as f:
        for line in f.readlines():
            if line.strip() == md5_for_check:
                return True  # 已存在
    return False  # 不存在

# 3. 如果不存在，处理后记录
if not check_md5_hex(md5_hex):
    # 处理文件...
    save_md5_hex(md5_hex)  # 追加到 md5.text

# md5.text 文件格式：
# d41d8cd98f00b204e9800998ecf8427e
# 098f6bcd4621d373cade4e832627b4f6
# ...
```

#### MD5 的特性
- **唯一性**：不同文件几乎不可能有相同的 MD5
- **稳定性**：同一文件每次计算的 MD5 都相同
- **快速性**：计算速度快，即使是大文件

### 5.6 中间件系统设计

#### 中间件的作用
在智能体执行的关键节点插入自定义逻辑，类似"关卡检查站"。

#### 三种中间件对比

| 中间件 | 触发时机 | 典型用途 |
|--------|----------|----------|
| `@wrap_tool_call` | 每次调用工具前后 | 日志记录、权限检查、参数验证 |
| `@before_model` | 每次调用 LLM 前 | 状态记录、上下文注入 |
| `@dynamic_prompt` | 每次构建提示词前 | 场景切换、A/B 测试 |

#### 扩展示例：添加权限检查
```python
@wrap_tool_call
def auth_check(request: ToolCallRequest, handler: Callable):
    """检查用户是否有权限调用敏感工具"""
    sensitive_tools = ["fetch_external_data"]
    
    if request.tool_call['name'] in sensitive_tools:
        user_id = get_current_user()
        if not has_permission(user_id):
            raise PermissionError("无权访问")
    
    return handler(request)
```

   - 演示不同类型的用户查询
   - 展示报告生成功能

3. **动手实践**

   - 修改提示词观察效果变化
   - 调整向量检索参数看准确率变化

### 6.2 常见问题解答

**Q1: 为什么选择 Chroma 而不是其他向量库？**  
A: Chroma 轻量级、易上手，适合学习和小规模项目。生产环境可以考虑 Pinecone、Milvus 等。

**Q2: 如何提高检索准确率？**  
A: 
- 优化分片策略（调整 chunk_size 和 overlap）
- 使用更好的 embedding 模型
- 添加元数据过滤
- 使用混合检索（向量 + 关键词）

**Q3: ReAct 模式和普通对话有什么区别？**  
A: ReAct 可以主动调用工具获取实时信息，而普通对话只能依赖训练数据。

**Q4: 如何评估智能体的性能？**  
A: 
- 人工评估：回答准确性、完整性
- 自动化测试：构造标准问题集，统计正确率
- 工具调用效率：是否选择了合适的工具

**Q5: 生产环境需要考虑哪些问题？**  
A: 
- 并发控制和限流
- 错误处理和重试机制
- 日志和监控
- 成本控制（token 用量）
- 数据安全（用户隐私保护）
