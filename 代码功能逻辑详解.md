### 2.1 `utils/path_tool.py` — 路径工具

文件：[`utils/path_tool.py`](utils/path_tool.py)

**解决的问题：** 程序运行时，"当前目录"可能不是你想象的位置。如果直接用相对路径找文件（如 `config/rag.yml`），经常会报"文件不存在"。

**怎么解决：** 通过计算项目根目录的绝对路径，确保无论从哪里启动程序，都能正确找到文件。

#### 2.1.1 `get_project_root()` 函数（第 18-28 行）

```python
def get_project_root() -> str:
    """
    获取项目根目录的绝对路径

    原理: 本文件在 utils/path_tool.py，往上两级就是项目根目录
          utils/path_tool.py  →  utils/  →  项目根目录
    """
    current_file = os.path.abspath(__file__)        # 本文件的绝对路径
    current_dir = os.path.dirname(current_file)      # utils/ 目录
    project_root = os.path.dirname(current_dir)      # 项目根目录
    return project_root
```

**逐行解释：**

- `os.path.abspath(__file__)` —— `__file__` 是当前 Python 文件自身的路径，`abspath` 把它转成绝对路径，例如 `D:\AI大模型RAG与智能体开发_Agent项目\utils\path_tool.py`。
- 第一次 `dirname` —— 去掉文件名，得到 `D:\AI大模型RAG与智能体开发_Agent项目\utils`。
- 第二次 `dirname` —— 再去掉一层，得到 `D:\AI大模型RAG与智能体开发_Agent项目`，这就是项目根目录。

#### 2.1.2 `get_abs_path()` 函数（第 31-41 行）

```python
def get_abs_path(relative_path: str) -> str:
    """把相对路径转换成绝对路径"""
    return os.path.join(get_project_root(), relative_path)
```

**调用示例：**

```python
get_abs_path("config/rag.yml")
# 返回：D:\AI大模型RAG与智能体开发_Agent项目\config\rag.yml
# 无论从哪个目录启动 Python，结果都一样
```

**其他模块怎么用：**

```python
from utils.path_tool import get_abs_path
config_path = get_abs_path("config/rag.yml")   # 总是返回正确的绝对路径
```

### 2.2 `utils/logger_handler.py` — 日志系统

文件：[`utils/logger_handler.py`](utils/logger_handler.py)

**解决的问题：** `print()` 调试太原始，无法区分信息的重要程度，也无法持久化保存。

**怎么解决：** 用 Python 标准库 `logging` 创建统一的日志系统，日志**同时输出到两个地方**：

```
┌─── 控制台（终端窗口）→ 方便开发时实时看（INFO 及以上）
└─── 日志文件（logs/）→ 方便出问题时回头查（DEBUG 及以上，更详细）
```

#### 2.2.1 日志格式和目录（第 25-31 行）

```python
LOG_ROOT = get_abs_path("logs")
os.makedirs(LOG_ROOT, exist_ok=True)     # 确保 logs/ 目录存在

LOG_FORMAT = logging.Formatter(
    '%(asctime)s - %(name)s - %(levelname)s - %(filename)s:%(lineno)d - %(message)s'
)
```

**格式说明：**

| 字段 | 含义 | 示例 |
|------|------|------|
| `%(asctime)s` | 时间戳 | `2025-06-15 14:23:45,123` |
| `%(name)s` | 日志器名称 | `agent` |
| `%(levelname)s` | 日志级别 | `INFO` / `ERROR` |
| `%(filename)s:%(lineno)d` | 文件名:行号 | `rag_service.py:22` |
| `%(message)s` | 日志内容 | `服务启动成功` |

**输出示例：**

```
2025-06-15 14:23:45 - agent - INFO - rag_service.py:22 - 服务启动成功
2025-06-15 14:23:46 - agent - ERROR - vector_store.py:50 - 向量库连接失败
```

#### 2.2.2 `get_logger()` 函数（第 34-74 行）

```python
def get_logger(
    name: str = "agent",
    console_level: int = logging.INFO,
    file_level: int = logging.DEBUG,
    log_file: str = None,
) -> logging.Logger:
    logger = logging.getLogger(name)
    logger.setLevel(logging.DEBUG)

    if logger.handlers:    # 避免重复添加 Handler
        return logger

    # 控制台 Handler
    console_handler = logging.StreamHandler()
    console_handler.setLevel(console_level)
    console_handler.setFormatter(LOG_FORMAT)
    logger.addHandler(console_handler)

    # 文件 Handler（按日期自动生成文件名）
    if not log_file:
        log_file = os.path.join(LOG_ROOT, f"{name}_{datetime.now().strftime('%Y%m%d')}.log")

    file_handler = logging.FileHandler(log_file, encoding='utf-8')
    file_handler.setLevel(file_level)
    file_handler.setFormatter(LOG_FORMAT)
    logger.addHandler(file_handler)

    return logger
```

**关键设计：**

- `if logger.handlers: return logger` —— 防止重复添加 Handler（每次调用 `get_logger("agent")` 时返回同一个配置好的 logger）。
- 控制台只显示 `INFO` 及以上（减少噪音），文件记录 `DEBUG` 及以上（保留所有细节）。
- 日志文件按日期命名，如 `logs/agent_20250615.log`。

#### 2.2.3 导出默认 logger（第 78 行）

```python
logger = get_logger()   # 模块加载时自动创建一个名为 "agent" 的默认 logger
```

**其他模块怎么用：**

```python
from utils.logger_handler import logger
logger.info("服务启动成功")       # 控制台 + 文件都记录
logger.debug("向量维度: 1536")   # 只写文件，不在控制台显示
```

### 2.3 `utils/config_handler.py` — 配置加载

文件：[`utils/config_handler.py`](utils/config_handler.py)

**解决的问题：** 项目中有很多可调节的参数（模型名称、数据库路径、分片大小等），如果散落在各个文件里，修改时很难找。

**怎么解决：** 把所有参数集中到 `config/` 目录下的 YAML 文件里，用本模块统一加载。

#### 2.3.1 `_load_yaml()` 函数（第 18-30 行）

```python
def _load_yaml(filename: str) -> dict:
    """加载 config 目录下的 YAML 配置文件"""
    filepath = get_abs_path(filename)
    with open(filepath, "r", encoding="utf-8") as f:
        return yaml.load(f, Loader=yaml.FullLoader)
```

**工作流程：**

1. `get_abs_path(filename)` —— 把 `config/rag.yml` 转成绝对路径
2. `open(...)` —— 以 UTF-8 编码打开文件
3. `yaml.load(..., Loader=yaml.FullLoader)` —— 解析 YAML 内容为 Python 字典

#### 2.3.2 `get_api_key()` 函数（第 33-48 行）

```python
def get_api_key() -> str:
    """
    获取阿里云 DashScope API Key（统一入口）
    按优先级依次检查三个环境变量，取第一个有值的
    """
    return (
        os.getenv("DASHSCOPE_API_KEY")
        or os.getenv("OPEMAO_API_KEY")
        or os.getenv("ALIYUN_API_KEY")
```

**设计意图：** 不同开发者可能用不同的环境变量名存放 API Key，这个函数兼容三种命名，任意一个有值即可。未找到时返回空字符串（模型初始化时会报错提示）。

## 4. 第三层：数据处理（文档 → 可检索的知识）

### 4.1 `utils/file_utils.py` — 文件工具

文件：[`utils/file_utils.py`](utils/file_utils.py)

#### 4.1.1 `get_file_md5_hex()` — 计算文件指纹（第 9-37 行）

```python
def get_file_md5_hex(filepath: str) -> str | None:
    if not os.path.exists(filepath):
        logger.error(f"[MD5计算] 文件不存在: {filepath}")
        return None

    if not os.path.isfile(filepath):
        logger.error(f"[MD5计算] 路径不是文件: {filepath}")
        return None

    md5_obj = hashlib.md5()
    chunk_size = 4096  # 4KB 分片，避免大文件占用过多内存

    try:
        with open(filepath, "rb") as f:
            while chunk := f.read(chunk_size):
                md5_obj.update(chunk)
        return md5_obj.hexdigest()
    except Exception as e:
        logger.error(f"[MD5计算] 失败: {filepath}, 错误: {e}")
        return None
```

**什么是 MD5？** 就像文件的"身份证号"，一段固定的 32 位十六进制字符串（如 `d41d8cd98f00b204e9800998ecf8427e`）。文件内容不同，MD5 就不同。

**关键设计：**

- `chunk_size = 4096` —— 每次只读 4KB，用 `while` 循环分片读取。即使文件有 1GB，也不会爆内存。
- `while chunk := f.read(chunk_size)` —— "海象运算符"，既读取数据又判断是否读完（读到空内容时循环结束）。
- `hexdigest()` —— 返回 MD5 的十六进制字符串。

**调用示例：**

```python
get_file_md5_hex("data/扫地机器人100问.pdf")
# 返回：'a3f5c8e92d1b4a6c8e7f2d3b5a9c1e4f'
```

**用途：** 判断一个文件是否已经处理过（去重）。

#### 4.1.2 `list_files_by_extension()` — 按类型列文件（第 40-60 行）

```python
def list_files_by_extension(path: str, allowed_extensions: tuple[str]) -> tuple[str]:
    if not os.path.isdir(path):
        logger.error(f"[文件列表] 路径不是目录: {path}")
        return ()

    files = []
    for filename in os.listdir(path):
        if filename.lower().endswith(allowed_extensions):
            files.append(os.path.join(path, filename))

    return tuple(files)
```

**调用示例：**

```python

# 返回：('data/扫地机器人100问.pdf', 'data/故障排除.txt', ...)
```

### 4.2 `rag/multimodal_ocr.py` — 多模态 OCR 服务

文件：[`rag/multimodal_ocr.py`](rag/multimodal_ocr.py)

**解决的问题：** 有些文档是扫描件或图片，里面虽然有文字但不是"可选中"的文本。需要"看图识字"。

**怎么解决：** 调用通义视觉大模型（qwen-vl-max），让 AI "看"图片并把文字写出来。

#### 4.2.1 OCR 指令（第 23-30 行）

```python
OCR_PROMPT = """请精准识别这张图片中的所有文字内容，要求：
1. 保持原文的段落结构和顺序
2. 如果是手写体，请尽力识别每个字
3. 如果图片中有表格，请用Markdown表格格式输出
4. 如果图片中有公式，请用LaTeX格式输出
5. 只输出识别出的文字，不要添加任何解释或说明

识别结果："""
```

这段指令会作为提示词发送给视觉模型，约束它**只输出识别结果**，不要加"这是..."等废话。

#### 4.2.2 初始化视觉模型（第 36-38 行）

```python
def __init__(self):
    model_name = rag_conf.get("vision_model_name", "qwen-vl-max")
    self.vision_model = ChatTongyi(model=model_name, dashscope_api_key=get_api_key())
```

**默认模型：** `qwen-vl-max`（通义千问视觉大模型），能识别图片中的文字、表格、公式。

#### 4.2.3 `recognize_image()` — 识别单张图片（第 45-61 行）

```python
def recognize_image(self, image_path: str) -> Optional[str]:
    try:
        image_url = self._image_to_base64(image_path)   # 图片 → base64 字符串
        message = HumanMessage(content=[
            {"type": "text", "text": OCR_PROMPT},                        # 文字指令
            {"type": "image_url", "image_url": {"url": image_url}},      # 图片数据
        ])
        text = self.vision_model.invoke([message]).content.strip()
        if text:
            logger.info(f"[OCR] {image_path} 识别 {len(text)} 字")
            return text
        logger.warning(f"[OCR] {image_path} 无文字")
        return None
    except Exception as e:
        logger.error(f"[OCR] {image_path} 失败: {e}", exc_info=True)
        return None
```

**流程：**

1. `_image_to_base64(image_path)` —— 把图片文件读成 base64 编码的字符串（如 `data:image/jpeg;base64,/9j/4AAQSkZJRg...`）。这是为了让图片能通过文本协议传给 API。
2. 构造 `HumanMessage` —— LangChain 的消息对象，同时包含文字指令和图片数据。
3. `self.vision_model.invoke([message])` —— 把消息发送给视觉模型，模型返回识别结果。
4. `.content.strip()` —— 提取返回的文本内容，去掉首尾空白。

#### 4.2.4 `recognize_pdf_pages()` — 扫描版 PDF 识别（第 63-96 行）

```python
def recognize_pdf_pages(self, pdf_path: str, pages: Optional[List[int]] = None) -> List[Tuple[int, str]]:
    from pdf2image import convert_from_path
    images = convert_from_path(pdf_path)                # PDF 每页转成图片
    if pages:
        images = [images[i - 1] for i in pages if 0 < i <= len(images)]

    results = []
    with tempfile.TemporaryDirectory() as tmp:
        for i, image in enumerate(images):
            page_num = pages[i] if pages else (i + 1)
            img_path = os.path.join(tmp, f"page_{page_num}.jpg")
            image.save(img_path, "JPEG")                 # 保存为临时图片
            text = self.recognize_image(img_path)        # 调用单图识别
            if text:
                results.append((page_num, text))         # 收集 (页码, 文字)

    return results
```

**流程：** PDF → 每页转图片 → 逐页 OCR → 返回 `[(页码, 文字), ...]`

```python
 (2, "如何连接 Wi-Fi..."),
 (5, "故障代码 E3 表示...")]
```

**解决的问题：** 用户可能上传各种格式的文档（PDF、Word、图片、TXT），每种格式的解析方式不同。

#### 4.3.1 `load_pdf()` — 加载 PDF（第 18-54 行）

```python
def load_pdf(filepath: str, password: Optional[str] = None) -> list[Document]:
    documents = []

    # 1. 提取文本层（如果 PDF 有原生文字）
    try:
        text_docs = PyPDFLoader(filepath, password).load()
        if text_docs:
            logger.info(f"[PDF] {filepath} 提取文本层: {len(text_docs)}页")
            documents.extend(text_docs)
    except Exception as e:
        logger.warning(f"[PDF] {filepath} 文本提取失败: {e}")

    # 2. 提取并 OCR 图片（扫描版 PDF 走这条路）
    try:
        image_docs = _extract_images_from_pdf(filepath)
        if image_docs:
            documents.extend(image_docs)
    except Exception as e:
        logger.warning(f"[PDF] {filepath} 图片提取失败: {e}")

    return documents
```

**为什么要走两条路：**

- 有些 PDF 是"电子版"，里面文字可以直接提取（速度快、准确）。
- 有些 PDF 是"扫描版"，文字在图片里，必须用 OCR 识别。
- 本函数**两种都试**，结果合并返回，确保不丢内容。

**返回的 `Document` 对象：**

```python
Document(
    page_content="扫地机器人使用前请先充电...",   # 提取出的文本
    metadata={"source": "data/xxx.pdf", "page": 1}  # 来源文件和页码
)
```

#### 4.3.2 `load_docx()` — 加载 Word（第 86-133 行）

```python
def load_docx(filepath: str) -> list[Document]:
    from docx import Document as DocxDocument
    doc = DocxDocument(filepath)

    documents = []

    # 1. 提取段落文字
    text_parts = [para.text for para in doc.paragraphs if para.text.strip()]
    if text_parts:
        full_text = "\n\n".join(text_parts)
        documents.append(Document(
            page_content=full_text,
            metadata={"source": filepath, "type": "docx_text"}
        ))

    # 2. 提取并 OCR 内嵌图片
    image_docs = _extract_images_from_docx(filepath)
    if image_docs:
        documents.extend(image_docs)

    return documents
```

**Word 文档里的图片怎么处理：** `_extract_images_from_docx()` 会把 `.docx` 当成 zip 包打开（Word 文档本质是 zip），从 `word/media/` 目录提取图片，逐张调用 `ocr_service.recognize_image()` 识别。

#### 4.3.3 `load_image()` 和 `load_txt()`（第 203-250 行）

```python
def load_image(filepath: str) -> list[Document]:

    return [Document(page_content=text, metadata={"source": filepath, "type": "image_ocr"})]
```

| `load_docx` | `.docx` | python-docx + OCR | 内嵌图调用 |
| `load_image` | `.jpg/.png` | OCR 直接识别 | 必然调用 |
| `load_txt` | `.txt` | TextLoader | 不调用 |

**`Chroma` 是什么：** 开源的本地向量数据库，把文字转成向量后存入磁盘，支持语义检索。

**`RecursiveCharacterTextSplitter` 工作原理：**

```
原始长文档（1000 字）
    ↓ 先尝试按 "\n\n"（段落）切分
    ↓ 段落仍太长，再尝试按 "\n"（换行）切分
    ↓ 还太长，按 "."（句号）切分
    ↓ ...
最终得到多个 ≤200 字的小段，相邻两段重叠 20 字
```

#### 5.1.3 MD5 去重（第 57-75 行）

```python
def _check_md5_exists(self, md5_hex: str) -> bool:
    """检查文件的 MD5 是否已经入库过"""
    md5_file = get_abs_path(chroma_conf["md5_hex_store"])
    if not os.path.exists(md5_file):
        open(md5_file, "w", encoding="utf-8").close()
        return False
    with open(md5_file, "r", encoding="utf-8") as f:
        for line in f:
            if line.strip() == md5_hex:
                return True
    return False

def _save_md5(self, md5_hex: str):
    """入库成功后记录 MD5"""
    md5_file = get_abs_path(chroma_conf["md5_hex_store"])
    with open(md5_file, "a", encoding="utf-8") as f:
        f.write(md5_hex + "\n")
```

**`md5.text` 文件内容示例：**

```
a3f5c8e92d1b4a6c8e7f2d3b5a9c1e4f
d41d8cd98f00b204e9800998ecf8427e
b7f2a9e3c1d5f8a6b4e2c9d7a1f3e5b8
```

每行一个 MD5，代表一个已入库的文件。下次运行入库时，如果文件的 MD5 已存在于这个文件中，就跳过。

#### 5.1.4 文档入库主流程（第 79-132 行）

```python
def load_documents(self):
    # 1. 确定要处理哪些文件类型
    supported_exts = tuple(get_supported_extensions())   # ('.pdf', '.docx', '.txt', ...)

    # 2. 列出 data/ 下所有符合条件的文件
    data_dir = get_abs_path(chroma_conf["data_path"])
    file_list = list_files_by_extension(data_dir, supported_exts)

    # 3. 逐个处理文件
    for filepath in file_list:
        md5_hex = get_file_md5_hex(filepath)             # 计算 MD5
        if self._check_md5_exists(md5_hex):              # 检查是否已入库
            logger.info(f"[知识库] 跳过: {filepath} (已处理)")
            continue

        documents = load_document(filepath)              # 加载文档内容
        split_docs = self.splitter.split_documents(documents)  # 切成小段
        self.vector_store.add_documents(split_docs)      # 存入向量库（自动转向量）
        self._save_md5(md5_hex)                          # 记录 MD5
```

**完整流程：**

```
data/ 文件夹
    ↓
扫描支持的文件类型（.txt, .pdf, .docx, .jpg 等）
    ↓
计算 MD5，检查是否已入库过（去重）
    ↓
调用 document_loader 加载文档内容
    ↓
用文本分片器把长文档切成小段（每段约 200 字）
    ↓
存入 Chroma 向量数据库（自动转成向量）
    ↓
记录 MD5（下次跳过此文件）
```

`|` 是 LangChain 的"管道"语法（借鉴了 Unix 管道），数据从左到右依次流过每个环节：

4. `StrOutputParser()` —— 从大模型返回的对象中提取纯文本字符串

#### 6.3.3 `rag_summarize()` — 完整 RAG 流程（第 46-68 行）

```python
def rag_summarize(self, query: str) -> str:
    # 第一步：检索相关文档
    context_docs = self.retriever_docs(query)

    # 第二步：把检索结果拼接成结构化的参考资料文本
    context_parts = []
    for i, doc in enumerate(context_docs, start=1):
        context_parts.append(
            f"【参考资料{i}】: 参考资料：{doc.page_content} | 参考元数据：{doc.metadata}"
        )
    context = "\n".join(context_parts)

    # 第三步：提交给 AI 模型生成回答
    return self.chain.invoke({"input": query, "context": context})
```

**完整流程：**

```
用户问题: "小户型适合哪些扫地机器人？"
    ↓
第一步：检索
  从向量库中找到最相关的 3 个文档片段
    ↓
第二步：组装
  把 3 个片段拼接成参考资料文本：
  【参考资料1】: ...内容... | 元数据: {...}
  【参考资料2】: ...内容... | 元数据: {...}
  【参考资料3】: ...内容... | 元数据: {...}
    ↓
第三步：生成
  把用户问题 + 参考资料填入 RAG 提示词模板
  提交给 AI 模型
    ↓
AI 根据参考资料生成准确回答
```

```python
@tool(description="通过 DuckDuckGo 在互联网上搜索实时信息，当本地知识库无法回答时使用此工具")
def web_search(query: str) -> str:
    try:
        ddgs = DDGS()                                     # 创建搜索客户端
        results = list(ddgs.text(query, max_results=5))   # 发起搜索请求
        if not results:
            return f"未找到关于「{query}」的结果"
        summaries = []
        for r in results[:5]:
            title = r.get("title", "")
            body = r.get("body", "")
            href = r.get("href", "")
            summaries.append(f"标题：{title}\n摘要：{body}\n链接：{href}")
        return "\n---\n".join(summaries)                  # 用分隔线连接
    except Exception as e:
        return f"搜索失败: {e}"
```

**逐行解释：**

- `DDGS()` —— DuckDuckGo Search 客户端，负责发起 HTTP 请求
- `ddgs.text(query, max_results=5)` —— 实际向 DuckDuckGo 服务器发请求，返回**迭代器**
- `list(...)` —— 把迭代器转成列表（最多 5 条结果）
- `r.get("title", "")` —— 安全地取字典值，没有该键就返回空字符串
- `"\n---\n".join(summaries)` —— 用分隔线把所有结果拼成一个长字符串

### 8.1 `agent/tools/middleware.py` — Agent 中间件

文件：[`agent/tools/middleware.py`](agent/tools/middleware.py)

**中间件是什么？** 就像快递分拣中心的检查站，包裹（数据）经过时可以做额外处理。本项目定义了 3 个中间件，分别在不同时机执行。

#### 8.1.1 `monitor_tool` —— 工具调用监控（第 23-53 行）

```python
@wrap_tool_call
def monitor_tool(
    request: ToolCallRequest,
    handler: Callable[[ToolCallRequest], ToolMessage | Command],
) -> ToolMessage | Command:
    tool_name = request.tool_call['name']    # 获取工具名，如 "web_search"
    tool_args = request.tool_call['args']    # 获取参数，如 {"query": "xxx"}

    logger.info(f"[tool monitor] 执行工具: {tool_name}")
    logger.info(f"[tool monitor] 传入参数: {tool_args}")

    try:
        result = handler(request)     # 真正执行工具
        logger.info(f"[tool monitor] 工具 {tool_name} 调用成功")

        # 特殊处理：报告生成场景，打标记
        if tool_name == "fill_context_for_report":
            request.runtime.context["report"] = True

        return result
    except Exception as e:
        logger.error(f"工具 {tool_name} 调用失败: {e}")
        raise
```

**`@wrap_tool_call` 装饰器：** LangChain 提供的中间件装饰器，表示"包住工具调用"。`handler(request)` 是真正执行工具的函数，在它的前后可以插入自定义逻辑。

**执行流程：**

```
Agent 决定调用 web_search(query="xxx")
    ↓
① monitor_tool 前置逻辑：
    logger.info("[tool monitor] 执行工具: web_search")
    logger.info("[tool monitor] 传入参数: {'query': 'xxx'}")
    ↓
② handler(request) 执行工具本身：
    web_search("xxx") → DDGS().text("xxx") → 返回结果
    ↓
③ monitor_tool 后置逻辑：
    logger.info("[tool monitor] 工具 web_search 调用成功")
    ↓
④ 如果工具名是 fill_context_for_report：
    request.runtime.context["report"] = True  # 打报告标记
    ↓
返回结果给 Agent
```

#### 8.1.2 `log_before_model` —— 模型执行前日志（第 56-68 行）

```python
@before_model
def log_before_model(state: AgentState, runtime: Runtime):
    messages = state['messages']
    logger.info(f"[log_before_model] 即将调用模型，带有 {len(messages)} 条消息")
    logger.debug(f"[log_before_model] 最后一条: {type(messages[-1]).__name__} | {messages[-1].content.strip()}")
    return None     # 返回 None 表示不修改状态，继续执行
```

**`@before_model` 装饰器：** 表示这个函数在每次大模型被调用**之前**执行。

**做了什么：**

- 记录当前有多少条消息在对话历史中
- 记录最后一条消息的类型（`HumanMessage` / `AIMessage` / `ToolMessage`）和内容
- 返回 `None` 表示"我不修改任何东西，继续正常执行"

#### 8.1.3 `report_prompt_switch` —— 动态切换提示词（第 71-85 行）

```python
@dynamic_prompt
def report_prompt_switch(request: ModelRequest) -> str:
    is_report = request.runtime.context.get("report", False)

    if is_report:
        return load_report_prompts()      # 报告生成场景的专用提示词
    return load_system_prompts()          # 日常对话的默认提示词
```

**`@dynamic_prompt` 装饰器：** 表示这个函数可以**动态决定使用哪个 system prompt**。返回的字符串会被用作本次大模型调用的 system prompt。

**执行时机：** 每次大模型被调用之前（在 `log_before_model` 之后）。

**完整联动：**

```
用户："生成我的 6 月使用报告"
    ↓
Agent 识别到报告意图，调用 fill_context_for_report
    ↓
monitor_tool 拦截，设置 context["report"] = True
    ↓
下一轮大模型调用前：
  log_before_model → 记录日志
  report_prompt_switch → 检测到 report=True，返回 report_prompt.txt 内容
    ↓
大模型在报告提示词指导下生成报告
```

**三个中间件执行顺序：**

| 时机 | 中间件 | 作用 |
|------|--------|------|
| 工具调用前 | `monitor_tool`（前置） | 记录工具名 + 参数 |
| 工具调用后 | `monitor_tool`（后置） | 记录成功/失败 + 打报告标记 |
| 模型调用前 | `log_before_model` | 记录消息数量 + 最后一条 |
| 模型调用前 | `report_prompt_switch` | 动态选择 system prompt |

### 9.1 `agent/react_agent.py` — ReAct 智能体

**这是整个系统的"大脑"，负责理解问题、规划行动、生成回答。**

**ReAct 是什么意思？** = Reasoning（推理） + Acting（行动）

```
ReAct 循环：

  用户提问
    ↓
  思考(Reasoning)：这个问题需要什么信息？需要调用什么工具？
    ↓
  行动(Acting)：调用工具获取信息
    ↓
  观察(Observation)：工具返回了什么结果？
    ↓
  再思考：信息够了吗？
    ├── 不够 → 继续调用其他工具（回到"行动"）
    └── 够了 → 整合所有信息，生成最终回答
```

```python
def _build_messages(self, history: List[Dict], current_query: str) -> List:
```

```python
def _stream_response(self, messages: List):
    input_dict = {"messages": messages}
    for chunk in self.agent.stream(
        input_dict, stream_mode="values", context={"report": False}
    ):
        latest = chunk["messages"][-1]
        if latest.content:
            yield latest.content.strip() + "\n"
```

**关键概念：**

- `self.agent.stream(...)` —— LangChain Agent 的流式输出方法，逐步产出内容
- `stream_mode="values"` —— 每次产出 Agent 状态的完整值（包括当前所有消息）
- `context={"report": False}` —— 初始上下文，`report` 标记用于中间件判断
- `yield` —— 生成器关键字，逐块返回内容（实现打字机效果）

### 10.3 `zs_mcp/tool_adapter.py` — 工具适配器

文件：[`zs_mcp/tool_adapter.py`](zs_mcp/tool_adapter.py)

**把 MCP 协议的工具格式转换成 LangChain 的 Tool 格式。**

```python
def mcp_tools_to_langchain(client, tools_info, name_prefix="", desc_prefix=""):
    langchain_tools = []
    for tool_info in tools_info:
        tool_name = tool_info.name
        exposed_name = f"{name_prefix}{tool_name}"
        description = f"{desc_prefix}{tool_info.description}"

        # 关键：定义异步函数，调用远程 MCP 服务
        async def call_remote(*, _client=client, _name=tool_name, **kwargs):
            return await _client.call_tool(_name, kwargs)

        langchain_tools.append(Tool(
            name=exposed_name,
            description=description,
            func=None,                  # 不用同步
            coroutine=call_remote,      # 用异步
        ))
    return langchain_tools
```

**调用链路：**

```
Agent 调用 web_search(query="扫地机器人")
    ↓
LangChain 执行 Tool.coroutine(**kwargs)
    ↓
执行 call_remote(_client=客户端, _name="web_search", query="扫地机器人")
    ↓
await _client.call_tool("web_search", {"query": "扫地机器人"})
    ↓
HTTP POST 到 http://localhost:8001/sse
    ↓
MCP Server 接收请求，执行 web_search 函数
    ↓
结果通过 SSE 返回
```

```python
class MultiServerMCPClient:
    async def connect_all(self):
        for server_key, config in self.server_configs.items():
            await self._connect_one(server_key, config)   # 逐个连接

    async def _connect_one(self, server_key, config):
        client = Client(transport=url, headers=headers)
        await client.__aenter__()
        tools_info = await client.list_tools()

        # 加前缀避免命名冲突
        server_tools = mcp_tools_to_langchain(
            client, tools_info,
            name_prefix=f"{server_key}_",       # local_python_web_search
            desc_prefix=f"[{name}] ",
        )
```

**为什么要加前缀：** 如果两个 MCP 服务都有 `web_search` 工具，不加前缀就会冲突。加了前缀后：

- 服务 B → `other_service_web_search`

```python
prompt = st.chat_input("请输入您的问题...")

if prompt:
    # 1. 保存用户消息
    st.session_state["messages"].append({"role": "user", "content": prompt})

    # 2. 显示用户消息
    with st.chat_message("user"):
        st.write(prompt)

    # 3. 调用 Agent 并流式获取回答
    response_container = st.chat_message("assistant")
    message_placeholder = response_container.empty()

    full_response = ""
    with st.spinner("思考中…"):
        res_stream = st.session_state["agent"].execute_stream_with_history(
            st.session_state["messages"][:-1], prompt
        )

        # 4. 逐块显示回答（打字机效果）
        for chunk in res_stream:
            full_response += chunk
            message_placeholder.markdown(full_response + "▌")   # ▌ 光标指示器

    # 5. 完成：移除光标，保存回答
    message_placeholder.markdown(full_response)
    st.session_state["messages"].append({"role": "assistant", "content": full_response})
```

**关键概念：**

- `st.chat_input(...)` —— 聊天输入框
- `st.chat_message("user"/"assistant")` —— 消息气泡容器
- `st.session_state["messages"]` —— 持久化的对话历史
- `message_placeholder.empty()` —— 创建占位符，可以反复更新内容
- `message_placeholder.markdown(full_response + "▌")` —— 每次有新内容时覆盖占位符，`▌` 模拟光标

## 14. 关键概念速查与常见问题排查

### 14.1 关键概念速查

| 概念 | 解释 |
|------|------|
| **RAG** | 检索增强生成：先从知识库找资料，再让 AI 根据资料回答 |
| **ReAct** | 推理+行动：AI 先思考、再调用工具、观察结果、继续思考 |
| **MCP** | 模型上下文协议：让 AI 能远程调用工具的标准接口 |
| **向量数据库** | 用"语义向量"存储文档，支持"意思相近"的智能搜索 |
| **Embedding** | 把文字转成数字向量的过程，语义相近的文字向量也相近 |
| **Prompt** | 给 AI 的指令，告诉它角色、规则和行为规范 |
| **中间件** | 在工具调用或模型执行前后插入自定义逻辑 |
| **Streamlit** | Python Web 框架，用纯 Python 就能做出交互式网页 |
| **LangChain** | AI 应用开发框架，提供模型调用、工具使用、链式处理等能力 |
| **Chroma** | 开源向量数据库，本地存储，支持语义检索 |
| **FastMCP** | MCP 协议的 Python 实现框架，简化 MCP Server/Client 开发 |
| **DDGS** | DuckDuckGo Search 的 Python 客户端，用于联网搜索 |

### 14.2 常见问题排查

| 症状 | 可能原因 | 修复 |
|------|---------|------|
| "文件不存在" 错误 | 相对路径错乱 | 检查 `get_abs_path` 调用 |
| 模型调用失败 | API Key 未配置 | 设置环境变量 `DASHSCOPE_API_KEY` |
| 向量库为空 | 未运行入库 | `python rag/vector_store.py` |
| 检索结果不相关 | chunk 太大或 embedding 模型问题 | 调小 `chunk_size`，检查 embedding 模型 |
| 工具调用失败 | 工具未注册或参数错误 | 检查 `local_tools` 列表和 `@tool` 装饰器 |
| Agent 不调工具 | 提示词未引导 | 检查 `main_prompt.txt` 中的工具说明 |
| 报告生成失败 | `fill_context_for_report` 未触发 | 检查中间件 `monitor_tool` |
| MCP 连接失败 | 服务未启动 | `python zs_mcp/start_robot_mcp_server.py` |
| 联网搜索失败 | 网络问题或缺依赖 | `pip install duckduckgo-search`，检查外网 |
| PDF OCR 失败 | 缺 `pdf2image` 或 `poppler` | 安装依赖和 poppler |
| Web 界面空白 | Agent 初始化异常 | 查看控制台日志，检查 API Key |
| 流式输出无效果 | 未正确使用 `yield` | 检查 `_stream_response` 和 `execute_stream` |
